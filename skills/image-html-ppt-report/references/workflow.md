# Image + HTML PPT Report Workflow

Use this reference when producing or explaining a reusable image-HTML report workflow.

## Core Idea

Instead of treating the browser page as a website, treat it as a slide runtime:

```text
numbered image pages + optional narration audio + static index.html controls
```

The browser only needs to display and navigate stable slide images. This keeps layout deterministic and makes the package easy to send to colleagues.

This workflow stops at slide images, optional narration audio, static HTML, and catalogs. Do not add video generation unless a separate skill or task asks for it.

PDF export should be generated by combining ordered slide images (`NN-topic.png`) directly, not by rendering `index.html` to PDF.

## Asset Pattern

Recommended page unit:

```text
NN-topic.md
NN-topic.png
NN-topic-script.md
NN-topic-script-edge.mp3
```

Use:

- `.md` for slide image prompts or page design instructions.
- `.png` for final slide visuals.
- `-script.md` for presenter script.
- `-script-edge.mp3` or similar for generated narration.

`NN-topic-script.md` must contain only narration text. The narration generator sends all text in the file to TTS. Do not include duration notes, tone guidance, headings, bullets, markdown annotations, or production comments. Put those details in the prompt file, outline, or a separate notes file.

The narration generator should not add spoken content by default; it only preserves paragraph breaks.

Only `.png` and `index.html` are required for a basic deck.

## Creation Modes

Use one of three modes.

### 1. Guided Creation

Best for formal reports or important internal presentations. The human reviews the important decisions as the deck is created.

Flow:

```text
topic/goal -> confirm
outline -> confirm
per-page prompts -> confirm when quality matters
slide images -> review
scripts -> review when narration matters
audio -> optional listening check
index.html -> automatic
```

### 2. Outline-Approved Generation

Best for drafts, internal material, or situations where the user trusts the style after approving the structure.

Flow:

```text
topic/goal -> confirm
outline -> confirm
prompts/images/scripts/audio/catalog/index.html -> automatic
```

After automatic generation, report any pages that need human review.

### 3. Assemble Only

Best when the user already has `NN-topic.png` files and optional scripts/audio.

Flow:

```text
scan folder -> flag blocking gaps -> generate index.html -> optional catalog
```

Use `scripts/build_index.py` for this mode.

## Player Pattern

For new decks, copy or generate from `assets/index-template.html`. The template already includes manual navigation, thumbnails, fullscreen, collapsible controls, optional audio mode, and keyboard shortcuts.

Use explicit arrays in the final HTML:

```js
const slides = ["01-title.png", "02-background.png"];
const audios = ["01-title-script-edge.mp3", "02-background-script-edge.mp3"];
```

This is more reliable than browser-side directory scanning because a local HTML file cannot normally list files in its folder.

Use `encodeURI(file)` when assigning file names to `img.src` or `audio.src`.

Keep one `render()` function responsible for:

- current image
- alt text
- filename or title display
- page counter
- previous/next disabled state
- audio source and progress reset
- transition class reset

Use one reusable `<audio>` element. On page change, pause current audio, set the new source, call `load()`, reset progress, and optionally play if autoplay mode is active.

## Suggested Controls

Good default controls:

- previous / next
- page counter
- thumbnail overview
- fullscreen
- collapsible toolbar
- manual/autoplay toggle
- audio play/pause
- audio progress

Thumbnail layout:

- Use the original slide images as thumbnails.
- Give each thumbnail card a fixed width instead of stretching to fill the row.
- Let cards wrap automatically for any slide count.
- Let the thumbnail panel scroll vertically when the wrapped grid exceeds the viewport height.

Good default shortcuts:

- left/right arrows
- space for next page
- `F` for fullscreen
- `O` for overview
- `C` for controls
- `A` for autoplay
- `P` for play/pause
- `Escape` for closing overlays

## Catalog Pattern

A reusable deck should include a Markdown catalog:

1. What the material is for.
2. Asset statistics by extension.
3. Page-by-page mapping.
4. HTML module breakdown.
5. Production workflow.
6. Naming rules.
7. Reuse checklist.
8. Known issues and future improvements.

## Packaging Guidance

When sharing a finished report:

- Send the whole report folder, not only `index.html`.
- Keep all referenced image/audio files beside `index.html` unless paths were intentionally organized into subfolders.
- Test by opening `index.html` from the packaged folder.
- Avoid absolute local paths in HTML.

When sharing this skill with colleagues:

- Share the skill folder itself: `image-html-ppt-report/`.
- The receiver can place it under their Codex skills directory or inside a project-local `.codex/skills/` folder.
- Keep example deck assets outside the skill unless they are intentionally maintained as templates.

## Common Risks

- The HTML references a file that was renamed after generation.
- Image and audio arrays have different lengths.
- Chinese filenames are not URI-encoded before loading.
- Audio autoplay fails because the browser requires user interaction.
- A deck works in the original folder but fails after packaging due to absolute paths.
- Thumbnail text or controls become too large on mobile.
